Diving into Optionals

In this article we will dive a little bit into Optionals.

@frozen enum Optional<Wrapped>

This took me by surprise a while back. Probably all seasoned Swift developers already know, but an Optional is just an enum.

An Optional is a type that represents either a wrapped value or nil, the absence of a value.

Since it is an enum, we could just declare it as Optional<MyClass> instead of as MyClass?

Actually we can also see how Optional is declared in the standard library:

enum Optional<Wrapped> : ExpressibleByNilLiteral {
    /// The absence of a value.
    ///
    /// In code, the absence of a value is typically written using the `nil`
    /// literal rather than the explicit `.none` enumeration case.
    case none

    /// The presence of a value, stored as `Wrapped`.
    case some(Wrapped)
}

We could dive even a bit further and check what ExpressibleByNilLiteral is, which is a type that can be initialised using the nil literal. Only the Optional type conforms to it and it is discouraged to be used in other places.

This protocol hides unnecessary implementation details. It is what lets us assign nil to an optional. There are more ExpressibleBy protocols out there. To dive more into it I recommend you to read this article:

Swift ExpressibleBy protocols

.none

Optional<Wrapped>.none is equivalent to the nil literal. The difference is that nil is a constant having .none as value. To go even further we can say that nil is of type Optional<Wrapped> .

Optional Binding

Optionals are a great to avoid cases in which you are dealing with nil. Most of the times we will conditionally bind the wrapped value of an Optional into a new variable with

if let, guard let, and switch

if let someConstant = someOptional {
    print(someConstant) 
} else {
    print("someOptional is nil...")
}

If we want to avoid the if else statement and we just want to unwrap our optional we can always do the following:

guard let someConstant = someOptional else { return }
print(someConstant)
// ...

Or even with a switch statement

switch someOptional {
    case let someConstant:
        print(someConstant)
    default:
        print("someOptional is nil...")
}

Optional Chaining

If we want to access the value of an optional without crashing our code we can use the postfix optional chaining operator or ?

var str:Optional<String> = "Hello World"
print(str.uppercased())

This code will fail with the following error message: Value of optional type 'Optional<String>' must be unwrapped to refer to member 'uppercased' of wrapped base type 'String'

To fix it we can use ?

var str:Optional<String> = "Hello World"
print(str?.uppercased())

And now we get a warning: Expression implicitly coerced from 'String?' to 'Any' what happens is that we are printing Optional("HELLO WORLD"). To fix this we have three choices:

Provide a default value to avoid this warning Force-unwrap the value to avoid this warning Explicitly cast to 'Any' with 'as Any' to silence this warning
Providing a default value sounds like a great idea!

The Nil-Coalescing Operator

When printing HELLO WORLD we do not want it to be wrapped by Optional(). Moreover we want to get rid of our warning. We could use Optional binding for this, but there is a better way. The The Nil-Coalescing Operator or ??

var str:Optional<String> = "Hello World"
print(str?.uppercased() ?? "")

We use ?? to supply a default value in case the Optional instance is nil. It can even be chained:

print(str1 ?? str2 ?? str3 ?? "str and str2 and str3 are all nil")

Unconditional Unwrapping

! otherwise known as Force Unwrap.

This one will get most of your PR rejected, and will trigger long discussions in your team. Be careful and use it if you have a very good reason that cannot be solved with Optional Binding.

var str:Optional<String> = "Hello World"
print(str!)

Using ! on a nil instance will definitely crash your app with a runtime error.

map and flatMap

A favourite of mine, this one can scratch some heads. Optional defines map and flatMap and they are incredibly useful. Used properly they can greatly simplify your code and make it more concise.

func flatMap<U>(_ transform: (Wrapped) throws -> U?) rethrows -> U?
func map<U>(_ transform: (Wrapped) throws -> U) rethrows -> U?

flatMap should be used with a closure that returns an optional whereas map should be with a method that returns a non-optional value. Would map return an optional, it will wrap with an optional.

let meaningoflifeString: String? = "42"
let mapMeaningoflife = meaningoflifeString.map { Int($0) }
let flatMapMeaningoflife = meaningoflifeString.flatMap { Int($0) }
print("map \(mapMeaningoflife) or flat \(flatMapMeaningoflife)")
// prints map Optional(Optional(42)) or flat Optional(42)\n"
